# AI PR Code Reading Guidelines

本文档定义 AI 在本仓库中“查看/阅读代码”时必须遵守的规则。目标是减少误读架构、越层改动和无效重构。

## 1. 必须先读的上下文

在开始任何代码修改前，必须先阅读：

1. `AGENTS.md`
2. `docs/architecture.md`
3. 与任务相关的模块头文件（`.h`）和实现文件（`.c`/`.lua`）

如果任务涉及 Lua 与 C 交互，必须同时阅读 `Source/lua/api/wamr_api.lua`、`src/wamr_bridge.*`、`src/app/lua_bindings.*`。

## 2. 阅读范围最小要求

AI 不能只看单个文件后直接修改。至少要补齐以下范围：

1. 目标函数本体（当前文件）
2. 上游调用路径（谁调用它）
3. 下游依赖路径（它调用谁）
4. 同层替代实现（同目录下相关模块）

建议优先用符号搜索确认调用链，再决定修改点。

## 3. 分层约束（硬规则）

必须遵守以下边界，不得靠“便利”打破：

1. `lua_bindings` 只做 Lua/native 参数编解码，不写业务策略。
2. `runtime_service` 负责应用用例编排，不泄漏 Lua marshalling 或 WAMR 内部细节。
3. `game_backend` 是后端门面；WAMR runtime/module/session 细节在 `backend/runtime_session` 封装。
4. `third_party/*` 是依赖，不是业务扩展点（除非任务明确要求）。
5. 卡带来源以 `Data/<bundleID>/cart` 为准，不以 `Source/cart` 为真值。

## 4. 变更前的自检要求

在动手改代码前，AI 必须确认：

1. 这次改动是否跨层；若跨层，是否有明确必要性。
2. 是否会影响 `wamr_*` 兼容接口语义。
3. 是否会影响启动器关键流程：`Rescan`、`Reset`、`Dither`、输入映射、逐帧执行。
4. 是否存在更小改动面（优先最小可行修复，不顺手重构）。

## 5. 明确禁止的阅读/改动行为

1. 只凭文件名或目录名猜测职责，不读实现即修改。
2. 发现“看起来重复”的代码就直接合并，未验证行为等价。
3. 为了绕过边界新增 pass-through 包装，造成“假分层”。
4. 未追踪调用链就调整公共 API 或结构体字段。
5. 把第三方依赖代码当作修业务问题的首选修改点。

## 6. PR 说明中的“阅读证据”要求

AI 提交 PR 时，描述中应包含：

1. 阅读过的关键文件列表（按“入口 -> 服务 -> 后端 -> 依赖”顺序）。
2. 一段调用链摘要（改动点在调用链中的位置）。
3. 分层边界说明（为什么没有越层；若越层，必要性是什么）。
4. 手工验证步骤与结果（至少包含构建和 simulator smoke test）。

若缺少以上信息，PR 可视为上下文不足，需要补充。
